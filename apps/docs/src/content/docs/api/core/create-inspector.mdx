Constructs the ring-buffer inspector the runtime uses to record snapshots. The runtime calls this for you when you pass `inspectorBufferSize` to [`createRuntime`](/api/core/create-runtime/); you only need `createInspector` directly when building custom runtime infrastructure (e.g. a devtools bridge that wants direct access to `record` / `subscribe`).

The implementation is allocation-free on the hot path: O(1) `record`, fixed slot array, wrap-around index. `getBuffer()` materializes a newest-first array on demand.

## Import

```ts
import { createInspector } from '@triggery/core/inspect';
```

`createInspector` re-exports through the main `@triggery/core` entry for backwards compatibility, but the `@triggery/core/inspect` subpath is the canonical home in v0.10+ â€” it keeps the inspector code out of the main bundle for apps that don't reach for it. The same subpath also exports `createInspectorFactory()` (a thin wrapper that returns `(bufferSize) => createInspector(bufferSize)`) â€” pass it as `createRuntime({ inspector: createInspectorFactory() })` when you want the runtime to instantiate the inspector on demand, with no static dependency from the main entry.

## Signature

```ts
function createInspector(bufferSize: number): InspectorImpl;
```

## Parameters

| Param | Type | Description |
|---|---|---|
| `bufferSize` | `number` | Number of snapshots to keep. Clamped to a minimum of 1. |

## Returns

An `InspectorImpl` object:

| Method | Signature | Description |
|---|---|---|
| `record(snapshot)` | `(s: TriggerInspectSnapshot) => void` | Push a snapshot. Wraps around the buffer. Notifies subscribers. |
| `getBuffer()` | `() => readonly TriggerInspectSnapshot[]` | Newest-first array of the last N records. |
| `getLastForTrigger(id)` | `(triggerId: string) => Snapshot \| undefined` | Most recent snapshot per trigger id. |
| `subscribe(fn)` | `(fn) => () => void` | Listener fired on every `record`. Returns an unsubscribe. |
| `clear()` | `() => void` | Empty the buffer and the per-trigger map. |

## Examples

### Build a custom devtools bridge

```ts twoslash
import { createInspector, type TriggerInspectSnapshot } from '@triggery/core';

const inspector = createInspector(200);

inspector.subscribe((snapshot: TriggerInspectSnapshot) => {
  globalThis.postMessage(
    { source: 'triggery-bridge', snapshot },
    '*',
  );
});

// Feed it manually â€” the example assumes you wired up your own dispatch
// recording. The runtime created via `createRuntime` calls `record` for you.
inspector.record({
  triggerId: 'demo',
  runId: '1',
  eventName: 'ping',
  status: 'fired',
  durationMs: 0.2,
  executedActions: ['log'],
  snapshotKeys: [],
});
```

### Tap into latest-per-trigger

```ts twoslash
import { createInspector } from '@triggery/core';

const inspector = createInspector(50);
const latest = inspector.getLastForTrigger('chat:new-message');
console.log(latest?.status);
```

### Subscribe and unsubscribe

```ts twoslash
import { createInspector } from '@triggery/core';

const inspector = createInspector(50);

const unsubscribe = inspector.subscribe(snapshot => {
  console.log(snapshot.triggerId, snapshot.status);
});

// Later, when the panel unmounts:
unsubscribe();
```

## Notes

<Aside type="tip">
The inspector is the single source of truth for `runtime.subscribe`, `runtime.getInspectorBuffer`, `trigger.inspect()`, and the React `useInspect` / `useInspectHistory` hooks. Replacing the standard inspector is rare â€” usually you want the standard one and a custom `Middleware` that mirrors writes to your own buffer.
</Aside>

<Aside type="caution">
The inspector has no awareness of which runtime it belongs to. If you reuse one inspector across multiple runtimes you'll get interleaved snapshots â€” devtools panels typically want one inspector per runtime.
</Aside>
